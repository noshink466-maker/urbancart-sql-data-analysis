# UrbanCart Data Analysis (PostgreSQL)

## Overview

UrbanCart is a fictional Bangladesh-based e-commerce retailer selling everyday products such as groceries, electronics, and household items. This project simulates a real-world data analyst task: taking raw transactional data (customers, orders, order items, products, and payments) and turning it into actionable business insights using SQL.

The dataset is structured as a small star-schema data warehouse — one fact table for orders, one for order line items, one for payments, and two dimension tables for customers and products. Using PostgreSQL, this project answers 25 business questions covering sales performance, customer behavior, product trends, payment patterns, and operational reporting.

This project was built as part of a hands-on SQL learning exercise, covering everything from schema design and data import to writing intermediate/advanced SQL queries (joins, aggregations, window functions, subqueries, and self-joins).

---

## Dataset Overview

The dataset represents customer orders, products purchased, payment details, and customer demographics for UrbanCart.

| Table | Description | Rows |
|---|---|---|
| `dimcustomers` | Customer details (name, gender, email, city, signup date) | 100 |
| `dimproducts` | Product catalog (name, category, price, stock) | 41 |
| `factorders` | Order-level data (customer, date, status) | 1,200 |
| `factorderitems` | Line-item details per order (product, quantity) | 4,800 |
| `factpayment` | Payment method per order | 1,200 |

## Business Questions Answered

1. How many total orders has UrbanCart received so far?
2. Which cities generate the highest number of orders & revenue?
3. What percent of total customers use Gmail?
4. What is the monthly trend of total orders over time?
5. *(reserved)*
6. What is the total revenue generated by UrbanCart?
7. Which product categories contribute the most to total revenue?
8. Which individual products generate the highest revenue?
9. What is the Average Order Value (AOV) and Average Basket Size?
10. Which products are at risk of stock-out due to high sales volume and low inventory?
11. Which customers contribute the highest total revenue?
12. What is the cancellation rate by city and by customer?
13. Do male and female customers show different purchasing patterns by category?
14. How does customer purchasing behavior change over time since account creation?
15. Which customers ordered in October but not in December?
16. Which customers ordered in both October and December?
17. Which payment methods are used most frequently?
18. Is there a relationship between payment method and order status?
19. Do certain cities prefer specific payment methods?
20. Are higher-value orders associated with specific payment methods?
21. What is the average number of items per order by payment method?
22. What is the difference between each product's price and its category's average price?
23. Which product pairs are most frequently purchased together?
24. Which product pairs generate the highest combined revenue when sold together?
25. Generate a daily report: Date, Total Orders, Total Items, Completed Orders, Cancelled Orders, Total Revenue.

## Key Insights

- A small number of cities account for a disproportionate share of both order volume and revenue, highlighting where UrbanCart's customer base is concentrated.
- Certain product categories consistently outperform others in revenue, suggesting where inventory and marketing focus should be prioritized.
- Several products are selling faster than their current stock can support — early signals for restocking.
- Payment method choice varies meaningfully by city and order status, which could inform localized payment partnerships or fraud/cancellation monitoring.
- Customer purchasing behavior shifts over time after account creation, useful for retention and lifecycle marketing strategies.

## Skills Demonstrated

- Database schema design (fact & dimension tables, primary/foreign keys)
- Data import via pgAdmin (CSV → PostgreSQL)
- Multi-table JOINs (INNER JOIN, self-joins for product-pair analysis)
- Aggregate functions (`SUM`, `COUNT`, `AVG`) with `GROUP BY` and `HAVING`
- Window functions (`OVER (PARTITION BY ...)`)
- Subqueries and `FILTER` clauses for conditional aggregation
- Date/time functions (`EXTRACT`, `DATE_TRUNC`, `TO_CHAR`)
- Debugging real-world SQL errors (case sensitivity, column mismatches, foreign key constraints)

## How to Run This Project

1. Run `schema/create_tables.sql` in PostgreSQL to create all tables.
2. Import each CSV from `data/` into its matching table using pgAdmin's Import/Export tool (in dependency order: customers → products → orders → order items → payments).
3. Open `queries/analysis_queries.sql` and run each query to reproduce the analysis.
